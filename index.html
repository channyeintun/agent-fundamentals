Introduction

Think about the last time you asked Gemini or any other AI to do something for you. Maybe you asked it to help plan a trip, manage your calendar, or analyze some data. It probably gave you great advice, like detailed steps, helpful suggestions, or even some sample code. But then you still had to do all the actual work yourself.

This gap between knowing and doing represents the next significant step in the evolution of AI. While we have incredibly intelligent systems capable of advising us on a vast array of tasks, the final execution still rests on our shoulders. It’s akin to having a world-class advisor who can’t pick up the phone, send an email, or press a button on your behalf.

Let's trace how we got here and each step’s limitations.

First, we had LLM APIs. These were revolutionary. Suddenly we could have conversations with AI, ask questions, and generate content. But try asking an LLM to check your email or book a flight. It can’t. The LLM is frozen in time and disconnected from the world. It can process information but can’t interact with external systems.

Then came LLMs with function calling. This seemed like the answer! Now the model could call functions, interact with APIs, and even update databases. But here's the catch: you still had to orchestrate everything. Every function call needed your explicit instruction. Want to book a complete trip? You'd need to manually coordinate dozens of function calls: search flights, check availability, compare prices, book flights, find hotels, book hotels, and arrange transportation. The model became a better assistant, but you were still the one doing all the thinking and planning.

Here’s what this looks like in practice:
Imagine asking “book me a trip to Paris next week.”

With an LLM, you get advice: “Here’s how to book a trip to Paris…” With function calling, you get capability but not autonomy: “I can search for flights if you tell me the exact dates, then I can search for hotels if you tell me the area, then I can…” and so on.

What you really want is: “I’ve found three flight options that work with your calendar, matched them with nearby hotels in your budget, and I’m ready to book the combination that works best. Should I proceed?”

That’s the promise of agents.

The solution:
Agent abstraction

Why agents are different

An agent isn’t just an LLM with tools bolted on. It’s a fundamentally different abstraction; a system that can pursue goals autonomously. Instead of responding to each request in isolation, an agent maintains context, plans ahead, and takes initiative to achieve objectives.

According to Google's agent whitepaper, in its most fundamental form:

“A generative AI agent can be defined as an application that attempts to achieve a goal by observing the world and acting upon it using the tools that it has at its disposal. Agents are autonomous and can act independently of human intervention, especially when provided with proper goals or objectives they are meant to achieve. Agents can also be proactive in their approach to reaching their goals.”

Five essential characteristics

Not everything that uses an LLM and tools is an agent. Breaking down this definition, true agents are characterized by five essential capabilities:

1

Goal-directed behavior

Agents work toward objectives, not just respond to queries. They understand the difference between the current state and the desired state, and they take actions to bridge that gap.

2

Autonomous operation

Agents can act independently of human intervention. Once given a goal, they determine the steps needed without requiring explicit instruction for each action.

3

Proactive initiative

Agents don’t just respond—they actively work toward goals. Even in the absence of explicit instruction sets, an agent can reason about what it should do next to achieve its ultimate goal.

4

Environmental awareness

Agents observe and perceive their environment through various inputs like user requests, API responses, database states, and error messages. They build a model of the world and update it as they work.

5

Tool use

Agents interact with the outside world using the tools at their disposal. This might include APIs, databases, external services, or other systems, enabling them to go beyond just generating text to actually performing actions.

The problem:
Building intelligence
without structure

Now that you understand what agents are and why they matter, let’s peek under the hood. How does something go from being a text-generating model to an autonomous problem-solver?

In module 1, you learned that agents are defined by five essential characteristics: goal-directed behavior, autonomous operation, proactive initiative, environmental awareness, and tool use.

Now here's the interesting part—all five of these characteristics emerge from just three architectural components working together.

According to Google’s agent whitepaper, every agent is built from these three components: the model (thinking), the tools (doing), and the orchestration (the process connecting them). It’s like a car—no matter how fancy the make, they all have an engine, wheels, and a steering system. These three components work together to create the autonomous, goal-directed behavior that defines an agent.

Think of it this way: The model enables autonomous decision-making and environmental awareness. The tools enable real-world action. The orchestration enables goal-directed, proactive behavior. Three components, working together, produce all five agent characteristics.

The hidden complexity of "Simple" tasks

What happens if you try to build an intelligent system without this structure? Imagine a user says: “Schedule a meeting with Sarah next week.” Your system needs to:

1 Understand who Sarah is.
2 Check both calendars for availability.
3 Find a mutually free time.
4 Send a meeting invitation.
5 Add it to both calendars.
6 Confirm with the user.

Without the proper three-component architecture, you end up writing specific code for every possible sequence soon you have thousands of lines of brittle if-then-else logic that breaks the moment someone asks something slightly different.

The root problem is trying to handle complexity with rigid code instead of flexible, intelligent behavior.

1. The Model (Thinking)

According to the whitepaper, the model serves as the centralized decision maker for all agent processes. It's the reasoning engine that enables the agent to understand goals, make decisions, and adapt based on feedback.

Understanding intent

Takes natural language and extracts the actual goal while handling ambiguity. It recognizes destination confusion (Paris, France vs Paris, Texas) and asks for clarification.

Making decisions

At each step, it decides what to do next—prioritizing flights vs hotels or asking for info vs using defaults based on dependencies.

Learning from feedback

When a tool returns results or errors, the model interprets the feedback and adjusts its approach (e.g., trying different dates for expensive flights).

2. Tools (Doing)

If the model is the brain, tools are the hands; they let the agent actually do things in the world. "Without tools, an agent is just a brain in a jar."

Pre-built integrations

Handle common tasks by connecting to services like Google Flights or weather APIs. They package complex logic into simple commands.

Custom functions

Allow the agent to leverage your existing application code. Perfect for operations requiring control or human approval before execution.

Information retrieval

Accesses company documentation, customer history, and catalogs to find the right information when needed rather than knowing everything upfront.

3. Orchestration (the process)

The orchestration layer is a cyclical process that governs how the agent takes in information, performs internal reasoning, and uses that reasoning to inform its next action or decision. This loop continues until the agent reaches its goal.

The Agent Loop

Perceive ➟ Think ➟ Act ➟ Check

The cycle repeats until the agent achieves its goal or realizes it needs user input.

Techniques like ReAct (reasoning + acting), Chain-of-thought, or Tree-of-thoughts help the model decide what to do at each step, but the core pattern remains: Perceive → Think → Act → Check.

What is RAG?

Retrieval-Augmented Generation (RAG) is a technique that gives an LLM access to external data not included in its training set. Even the most powerful models have a "knowledge cutoff"—they don't know what happened yesterday, and they certainly don't know your company's private data.

Think of an LLM as a brilliant student taking an exam. Without RAG, it's an "opened book" test where the student only relies on memory. With RAG, it's an "open book" test where the student can look up facts in a library before answering.

Why it matters for agents

RAG is a critical "environmental awareness" tool. It allows agents to:

  • • Reduce hallucinations: Grounding responses in actual documents.
  • • Provide up-to-date info: Accessing real-time news or databases.
  • • Handle private context: Reasoning over your specific emails, docs, or code.

The problem:
The right tool
for the wrong job

By now, it’s clear that agents are incredibly powerful. They can plan, use tools, and autonomously pursue goals. But there’s a trap many developers fall into: trying to use agents for everything.

When you have a hammer, everything looks like a nail. But as our agent whitepaper notes, using a sophisticated agent for a simple, deterministic task is like using a rocket ship to go to the grocery store—it’s expensive, slow, and prone to unnecessary failure.

The biggest challenge in building with agents isn’t the technical implementation—it’s knowing when not to use one. Over-engineering simple tasks with agents introduces latency, cost, and "hallucination risk" where none existed before.

Before you build, you need to ask: Does this task actually require autonomous reasoning and adaptation, or is it just a sequence of steps that can be handled more reliably with traditional code?

Three signs you don't need an agent

The task is simple and single-step

If you're just summarizing a single document or translating a sentence, you don't need an agent. A simple LLM API call is faster, cheaper, and more reliable.

The process is deterministic

If you know exactly which steps to take every time, use a workflow or script. Traditional code is 100% reliable; agents are not. Use agents only when the path isn't known upfront.

Latency is critical

The agent "Loop" is slow because it has to think, act, and check. If your users need a sub-second response, avoid agentic architectures unless they're running in the background.

"Just because you can use an agent doesn't mean you should."

The solution:
Clear decision criteria

So when should you use an agent? According to Google’s agent whitepaper, you should use an agent when your task meets three specific criteria: it requires multi-step reasoning, it needs to interact with external tools, and it must adapt to dynamic feedback.

The Sweet Spot

Agents are the right tool when you have a complex goal where the specific sequence of actions cannot be pre-defined because it depends on information found during the process.

Advice for getting started

Start simple

Don't build a complex agent from day one. Start with a single API call, then add tools, and finally add orchestration only where necessary.

Monitor closely

Agents are non-deterministic. You need extensive logging and monitoring to understand why an agent made a specific decision or where it got stuck.

Human in the loop

For high-stakes actions (sending money, deleting data), always include a step where a human must approve the agent's proposed action.

Define clear goals

An agent is only as good as its objective. Be precise in what you want the agent to achieve and what constraints it should respect.
